## 一、分层推理概述

分层推理（Hierarchical Reasoning）是指大模型根据任务复杂度，在不同"思考深度"级别之间切换的能力。它让模型既能对简单问题快速响应，又能对复杂问题展开深度思考，实现**效率与质量的动态平衡**。

### 为什么需要分层推理？

```
┌─────────────────────────────────────────────────────────────┐
│                    分层推理解决的问题                         │
├─────────────────────────────────────────────────────────────┤
│  ❌ 问题1：所有问题都用"全力思考" → 慢、贵、用户体验差        │
│  ❌ 问题2：所有问题都"直接回答" → 复杂任务出错率高            │
│  ✅ 解决：简单问题快答，复杂问题深想，动态适配               │
└─────────────────────────────────────────────────────────────┘
```

| 模型 | 推理层级 | 控制方式 | 核心机制 | 特色功能 |
|:-----|:---------|:---------|:---------|:---------|

## 三、DeepSeek-V4：三档推理强度

### 3.1 机制设计

DeepSeek-V4 提供**三个推理档位**，通过 `reasoning_effort` 参数控制：

| 档位 | 参数值 | 特点 | 适用场景 |
|:-----|:-------|:-----|:---------|
| **Non-think** | `thinking.type: "disabled"` | 快速直觉响应，无思维链输出 | 简单问答、闲聊、格式转换 |
| **Think High** | `reasoning_effort: "high"` | 有意识逻辑分析，默认档位 | 常规推理、数学计算、代码生成 |
| **Think Max** | `reasoning_effort: "max"` | 将推理推向极致，最慢最准 | 复杂Agent任务、Claude Code类场景 |

### 3.2 技术实现细节

```python
# DeepSeek-V4 API调用示例
from openai import OpenAI

client = OpenAI(api_key="<key>", base_url="https://api.deepseek.com")

# Think High模式（默认）
response = client.chat.completions.create(
    model="deepseek-v4-pro",
    messages=[{"role": "user", "content": "解这个方程：2x² + 5x - 3 = 0"}],
    reasoning_effort="high",  # high或max
    extra_body={"thinking": {"type": "enabled"}}
)

# 提取分离的输出
reasoning = response.choices[0].message.reasoning_content  # 思维链
answer = response.choices[0].message.content                  # 最终答案
```

**输出结构分离**：
- `reasoning_content`：内部思考过程（对用户不可见，但可提取）
- `content`：面向用户的最终回答

**特殊约束**：
- 思考模式下 `temperature`、`top_p`、`presence_penalty`、`frequency_penalty` **不生效**
- 工具调用场景必须完整回传 `reasoning_content`，否则返回400错误

### 3.3 实际例子：解方程

**问题**：`2x² + 5x - 3 = 0`，求x的值

#### Non-think模式（快速响应）
```
[content]: 
x = 0.5 或 x = -3

（无reasoning_content输出）
```
- 耗时：~0.5秒
- Token消耗：~15个

#### Think High模式（标准推理）
```
[reasoning_content]:
我需要解这个一元二次方程 2x² + 5x - 3 = 0。
使用求根公式：x = (-b ± √(b² - 4ac)) / 2a
其中 a = 2, b = 5, c = -3
判别式 Δ = b² - 4ac = 25 - 4×2×(-3) = 25 + 24 = 49
√49 = 7
x₁ = (-5 + 7) / 4 = 2/4 = 0.5
x₂ = (-5 - 7) / 4 = -12/4 = -3
验证：2×(0.5)² + 5×0.5 - 3 = 0.5 + 2.5 - 3 = 0 ✓

[content]:
方程 2x² + 5x - 3 = 0 的解为：

x₁ = 0.5
x₂ = -3

使用求根公式计算，判别式 Δ = 49 > 0，有两个不等实根。
```
- 耗时：~2秒
- Token消耗：~180个（reasoning: 120 + content: 60）

#### Think Max模式（极致推理）
```
[reasoning_content]:
【问题分析】
这是一个标准的一元二次方程：ax² + bx + c = 0
给定：2x² + 5x - 3 = 0

【方法选择】
可用方法：
1. 因式分解法
2. 求根公式法
3. 配方法

先尝试因式分解：
2x² + 5x - 3
需要找到两个数，乘积为 2×(-3) = -6，和为 5
这两个数是 6 和 -1
所以：2x² + 6x - x - 3 = 2x(x+3) - 1(x+3) = (2x-1)(x+3) = 0

验证因式分解：
(2x-1)(x+3) = 2x² + 6x - x - 3 = 2x² + 5x - 3 ✓

【求解】
(2x-1)(x+3) = 0
2x - 1 = 0 → x = 1/2 = 0.5
x + 3 = 0 → x = -3

【验证】
代入 x = 0.5: 2×(0.25) + 2.5 - 3 = 0.5 + 2.5 - 3 = 0 ✓
代入 x = -3: 2×9 + (-15) - 3 = 18 - 15 - 3 = 0 ✓

【结论】
方程有两个实数解：x = 0.5 和 x = -3

[content]:
方程 2x² + 5x - 3 = 0 的解法如下：

**方法一：因式分解法（推荐）**
2x² + 5x - 3 = (2x - 1)(x + 3) = 0
解得：x₁ = 1/2 = 0.5，x₂ = -3

**方法二：求根公式法**
Δ = b² - 4ac = 25 + 24 = 49
x = (-5 ± 7) / 4
x₁ = 0.5，x₂ = -3

**答案**：x = 0.5 或 x = -3
```
- 耗时：~5秒
- Token消耗：~450个（reasoning: 320 + content: 130）

## 四、Kimi K2.6：双模式 + 保留式思考

### 4.1 机制设计

Kimi K2.6 提供**两种推理模式**：

| 模式 | 模型/配置 | Temperature | 特点 |
|:-----|:----------|:------------|:-----|
| **Thinking** | `kimi-k2.6` + `thinking.type: "enabled"` | **固定1.0** | 完整思维链，深度推理 |
| **Instant** | `kimi-k2.6` + `thinking.type: "disabled"` | **0.6** | 低延迟快速响应 |

**核心创新：Preserve Thinking（保留式思考）**
- 多轮对话中可选择保留历史 `reasoning_content`
- 让模型"记住"之前的推理链条，避免重复思考
- 通过 `thinking.keep: "all"` 开启

### 4.2 技术实现细节

```python
# Kimi K2.6 API调用示例
from openai import OpenAI

client = OpenAI(api_key="<key>", base_url="https://api.moonshot.cn/v1")

# Thinking模式（深度推理）
response = client.chat.completions.create(
    model="kimi-k2.6",
    messages=[{"role": "user", "content": "分析这段代码的bug"}],
    temperature=1.0,  # Thinking模式固定1.0
    extra_body={
        "thinking": {
            "type": "enabled",
            "keep": "all"  # 保留历史思考
        }
    }
)

# Instant模式（快速响应）
response = client.chat.completions.create(
    model="kimi-k2.6",
    messages=[{"role": "user", "content": "你好"}],
    temperature=0.6,  # Instant模式推荐0.6
    extra_body={
        "thinking": {"type": "disabled"}
    }
)
```

**流式输出顺序**：
```
chunk 1-50:   reasoning_content（思考过程）
chunk 51+:    content（最终回答）
```
通过判断 `content` 字段是否出现来识别思考结束。

### 4.3 实际例子：代码Debug（多轮保留式思考）

**场景**：用户让AI调试一个Python函数，需要多轮交互

#### 第一轮：初始分析

**用户**：`这个函数有bug，帮我找一下：def factorial(n): return n * factorial(n-1)`

```
[reasoning_content]:
用户提供了一个递归函数求阶乘，让我分析bug。

代码：
def factorial(n):
    return n * factorial(n-1)

问题分析：
1. 这是一个递归函数，但没有基准情况（base case）
2. 当 n = 0 时，应该返回 1（0! = 1）
3. 当前代码会无限递归，导致 RecursionError
4. 还需要考虑 n < 0 的情况

修复方案：
添加基准情况：if n == 0: return 1
添加输入校验：if n < 0: raise ValueError

[content]:
这个递归函数缺少**基准情况（base case）**，会导致无限递归。

**Bug**：当 n 不断减1，永远不会停止，最终抛出 RecursionError。

**修复代码**：
```python
def factorial(n):
    if n < 0:
        raise ValueError("n must be non-negative")
    if n == 0:
        return 1
    return n * factorial(n - 1)
```
```

#### 第二轮：用户追问（开启Preserve Thinking）

**用户**：`那如果我想用迭代方式实现呢？`

**如果不保留思考**：
```
[reasoning_content]:
用户想要用迭代方式实现阶乘函数。
迭代法思路：
1. 初始化 result = 1
2. 从 1 循环到 n
3. 每次乘以当前数字
4. 返回 result

[content]:
迭代实现：
```python
def factorial(n):
    if n < 0:
        raise ValueError("n must be non-negative")
    result = 1
    for i in range(1, n + 1):
        result *= i
    return result
```
```

**开启Preserve Thinking后**：
```
[reasoning_content]:
【上下文回顾】
上一轮我分析了用户的递归阶乘函数，发现缺少base case，
并提供了修复方案。用户现在想要迭代版本。

【迭代法设计】
基于之前的分析，我需要：
1. 保留输入校验（n < 0的处理）
2. 用循环替代递归
3. 处理 n = 0 的情况（result初始为1，循环不执行，返回1）

代码结构：
- 校验 n >= 0
- result = 1
- for i in range(2, n+1): result *= i
- return result

【与递归版对比】
- 迭代版：无栈溢出风险，O(1)空间复杂度
- 递归版：代码简洁，但有栈深度限制

[content]:
迭代实现（基于之前的分析，保留输入校验）：

```python
def factorial(n):
    if n < 0:
        raise ValueError("n must be non-negative")
    result = 1
    for i in range(2, n + 1):
        result *= i
    return result
```

**相比递归版的优势**：
- 无栈溢出风险
- 空间复杂度 O(1)
- 可以处理更大的 n 值
```

**Preserve Thinking的价值**：
- 模型"记得"之前发现了缺少base case的问题
- 迭代版自动保留了输入校验逻辑
- 回答更连贯，避免了重复分析

## 五、GLM-5.1：轮级控制 + 交错式思考

### 5.1 机制设计

GLM-5.1 提供**最细粒度的推理控制**：

| 控制维度 | 参数 | 说明 |
|:---------|:-----|:-----|
| **全局开关** | `thinking.type: enabled/disabled` | 开启/关闭思考 |
| **保留式思考** | `thinking.clear_thinking: false` | 保留历史reasoning_content |
| **轮级控制** | 每轮独立设置 | 不同轮次可开/关思考 |
| **交错式思考** | 默认支持 | 工具调用间继续推理 |

**Coding Plan端点默认开启保留式思考**，标准API需手动设置。

### 5.2 技术实现细节

```python
# GLM-5.1 API调用示例
from openai import OpenAI

client = OpenAI(api_key="<key>", base_url="https://api.z.ai/api/paas/v4/")

# 标准API开启保留式思考
response = client.chat.completions.create(
    model="glm-5.1",
    messages=messages,
    extra_body={
        "thinking": {
            "type": "enabled",
            "clear_thinking": False  # False=保留历史思考
        }
    }
)

# 轮级控制示例：第一轮开思考，第二轮关思考
messages = [
    {"role": "user", "content": "设计一个数据库架构"}  # 复杂任务，开思考
]

# 第一轮：开启思考
response1 = client.chat.completions.create(
    model="glm-5.1",
    messages=messages,
    extra_body={"thinking": {"type": "enabled"}}
)

messages.append({
    "role": "assistant",
    "content": response1.content,
    "reasoning_content": response1.reasoning_content
})

messages.append({
    "role": "user", 
    "content": "好的，把刚才的表名改成中文"  # 简单任务，关思考
})

# 第二轮：关闭思考（快速响应）
response2 = client.chat.completions.create(
    model="glm-5.1",
    messages=messages,
    extra_body={"thinking": {"type": "disabled"}}  # 关闭思考，快速响应
)
```

### 5.3 实际例子：复杂Agent任务（交错式思考）

**场景**：让AI完成"查询北京天气并生成穿衣建议"的任务

#### 第一轮：规划 + 工具调用

**用户**：`今天北京天气怎么样？我该穿什么？`

```
[reasoning_content]:
用户想知道北京今天的天气，并需要穿衣建议。

任务分解：
1. 先获取北京今天的天气数据
2. 根据温度和天气状况给出穿衣建议

需要调用的工具：
- get_weather(city="北京")

[tool_call]:
调用 get_weather(city="北京")
```

#### 工具返回结果

```json
{"weather": "多云", "temp": "15°C", "wind": "北风3级", "humidity": "45%"}
```

#### 第二轮：基于工具结果继续推理（交错式思考）

```
[reasoning_content]:
【工具结果分析】
天气数据：
- 天气：多云
- 温度：15°C
- 风力：北风3级
- 湿度：45%

【穿衣建议推理】
15°C属于凉爽天气，需要考虑：
1. 基础层：长袖T恤或薄衬衫
2. 外层：薄外套或风衣（防风，北风3级）
3. 下装：长裤
4. 鞋子：常规皮鞋或运动鞋
5. 配件：可带件薄毛衣备用

多云天气紫外线不强，不需要特别防晒。
湿度45%适中，体感舒适。

[content]:
北京今天多云，气温15°C，北风3级。

**穿衣建议**：
- 👕 上身：长袖T恤/衬衫 + 薄外套或风衣
- 👖 下身：长裤
- 👟 鞋子：常规款即可
- 🧥 备用：带件薄毛衣，早晚温差大时加穿

今天体感凉爽舒适，适合外出！
```

**交错式思考的价值**：
- 模型在收到工具结果后**继续思考**，而非直接输出
- 可以基于天气数据做深度分析（温度+风力+湿度的综合判断）
- 推理过程与工具调用**交错进行**，而非一次性完成

## 六、MiniMax M2.7：推理开关与自进化

### 6.1 机制设计

MiniMax M2.7 的分层推理相对简洁：

| 模式 | 控制方式 | 特点 |
|:-----|:---------|:-----|
| **标准模式** | 默认 | 直接响应，适合简单任务 |
| **Reasoning模式** | `reasoning: true` | 输出推理Token，适合复杂任务 |

**独特之处**：模型具有**自进化能力**，可以在训练过程中自主优化推理策略。

### 6.2 实际例子

```python
# MiniMax M2.7 API调用（通过OpenRouter）
import requests

response = requests.post(
    "https://openrouter.ai/api/v1/chat/completions",
    headers={"Authorization": "Bearer <key>"},
    json={
        "model": "minimax/minimax-m2.7",
        "messages": [{"role": "user", "content": "解释量子纠缠"}],
        "reasoning": True  # 开启推理模式
    }
)
```

**标准模式**：
```
量子纠缠是量子力学中的一种现象，两个粒子可以相互关联，
无论相距多远，测量其中一个会瞬间影响另一个的状态。
```

**Reasoning模式**：
```
[reasoning]:
用户要求解释量子纠缠。这是一个复杂的物理概念，我需要：
1. 用通俗语言解释核心概念
2. 避免过度数学化
3. 可以举EPR佯谬或贝尔实验的例子
4. 澄清常见的误解（如超光速通信）

[answer]:
量子纠缠是量子力学中最神奇的现象之一...
```

## 七、四种机制对比总结

### 7.1 机制对比表

| 对比维度 | DeepSeek-V4 | Kimi K2.6 | MiniMax M2.7 | GLM-5.1 |
|:---------|:------------|:----------|:-------------|:--------|
| **层级数量** | 3级 | 2级 | 2级 | 2级+轮级控制 |
| **控制粒度** | 全局 effort | 全局 + 保留 | 全局 | **轮级独立** |
| **思维链分离** | ✅ 双字段 | ✅ 双字段 | ✅ 双字段 | ✅ 双字段 |
| **跨轮保留** | 工具调用必须 | `keep: "all"` | 未明确 | `clear_thinking: false` |
| **交错思考** | 支持 | 支持 | 支持 | **原生支持** |
| **温度控制** | 思考模式无效 | Thinking固定1.0 | 未明确 | 未明确 |
| **特色功能** | 3档effort | Preserve Thinking | 自进化 | 轮级开关 |

### 7.2 适用场景推荐

| 场景 | 推荐模型/模式 | 理由 |
|:-----|:--------------|:-----|
| **简单问答**（天气、时间） | DeepSeek Non-think / Kimi Instant | 最快响应，最低成本 |
| **常规推理**（数学题、代码） | DeepSeek Think High / Kimi Thinking | 平衡质量与速度 |
| **复杂Agent**（多步工具调用） | DeepSeek Think Max / GLM-5.1 | 最深推理，最高准确率 |
| **多轮编码Debug** | Kimi + Preserve Thinking | 保留推理链，避免重复 |
| **混合任务**（简单+复杂交替） | GLM-5.1 轮级控制 | 每轮独立开关，最灵活 |
| **长周期任务**（8小时+） | GLM-5.1 保留式思考 | 持续优化，策略自修正 |
